# KnowBrain Agent 产品进化指南

> 版本：V1.0 | 日期：2026-07-19 | 基于产品定位 + 用户感知双重视角

---

## 一、核心命题

**技术做了 10 分，用户感知只有 3 分。** 当前系统的问题不是能力不够，而是能力不可见。Agent 的反思/多步检索在后台静默完成，最终呈现就是一个输入框 + 一个答案，和通用 Chatbot 没有感知差异。

客户老板说的「多 Agent 才能体现 AI」，本质上要的不是架构复杂度，而是**感知到的智能层次**。

本指南回答两个问题：

1. **怎么让已有的 AI 能力被用户感知到？**（低成本增强）
2. **怎么在不变成「四不像」的前提下扩展 Agent 能力？**（产品边界）

---

## 二、产品定海神针：知识全生命周期 Agent

### 2.1 一切 Agent 能力的锚点

KnowBrain 的护城河是**企业私有知识管理**。所有 Agent 能力必须服务于这个核心，不越界做通用 AI 平台。

```
知识全生命周期：

  采集 ──→ 整理 ──→ 检索 ──→ 应用 ──→ 进化
  ✅已完成    ✅已完成    ✅已完成    ↓↓↓       ↓↓↓
                                   待加强      待加强
```

Agent 扩展聚焦在「应用」和「进化」两个环节，让知识从**静态存储**变成**动态资产**。

### 2.2 边界法则：做什么 vs 不做什么

| ✅ 做（知识全生命周期 Agent） | ❌ 不做（通用 AI 平台） |
|------------------------------|------------------------|
| 知识摘要 Agent：「把这三份合同政策整合成要点」— 基于已有文档 | 写作 Agent：「帮我写一篇演讲稿」 |
| 知识巡检 Agent：「哪些文档内容冲突」「哪些制度已过期」 | 翻译 Agent：「翻译成英文」 |
| 新人向导 Agent：「我是新入职 Java 开发，要了解什么」→ 自动组装文档 | 代码 Agent：「帮我写 Python 脚本」 |
| 知识对比 Agent：「A 制度和 B 制度有什么区别」 | 通用问答：「今天天气怎么样」 |
| 场景路由 Agent：IT 专家 + HR 专家（基于种子数据的角色 Prompt） | 多 Agent 并行协作框架（过度工程化） |

**底线**：Agent 的输出必须溯源到知识库中的文档，不凭空创造内容。

## 三、低成本「AI 感」提升方案

以下方案按投入产出比排序，优先实施前两项。

### 3.1 思考链可视化 ⭐⭐⭐（投入极低，感知极强）

**当前**：Agent 的多步检索 + 反思完全不可见，用户只看到一个 loading 然后出结果。

**改造后**：实时展示 Agent 的思考过程。

```
用户输入：「全职和外包远程办公政策有什么区别」

界面展示：
  ┌────────────────────────────────────────────┐
  │ 🤔 分析：这是一个对比类问题，需要分别检索    │
  │                                            │
  │ 🔍 搜索：「全职员工 远程办公政策」            │
  │    → 找到 5 篇相关文档                       │
  │                                            │
  │ 🔍 搜索：「外包人员 远程办公政策」            │
  │    → 找到 3 篇相关文档                       │
  │                                            │
  │ ✅ 信息完整，开始对比分析...                  │
  │                                            │
  │ ──────────────────────────────────────     │
  │                                            │
  │ 【对比结论】                                │
  │ 全职员工：支持远程办公，需申请...             │
  │ 外包人员：远程办公需项目主管审批...           │
  │ 📎 来源：《远程办公管理制度》《外包人员管理办法》│
  └────────────────────────────────────────────┘
```

**实现方式**：Agent 循环中每步推一个 SSE 状态事件（`thinking` 类型），前端渲染为过程卡片。不改变 Agent 逻辑，只是把已有的内部状态暴露出来。

```
SSE 事件序列：
  event: thinking
  data: {"type":"analyze","text":"这是一个对比类问题"}

  event: thinking
  data: {"type":"search","text":"全职员工 远程办公政策","hits":5}

  event: thinking
  data: {"type":"search","text":"外包人员 远程办公政策","hits":3}

  event: thinking
  data: {"type":"synthesize","text":"信息完整，开始对比分析"}

  event: token / done
  data: ...
```

**粒度建议**：每步 2-3 行，不暴露过长内部信息。客户看到「AI 在思考」即可，不需要看到「调用 searchKnowledge(query='x', topK=8)」这种技术细节。

**预估工时**：后端 2 天 + 前端 2 天。

### 3.2 场景路由 Agent ⭐⭐⭐（投入低，差异化明显）

**核心理念**：感知是「多 Agent 协作」，实现是「分类器 + 不同 System Prompt」。

```
用户输入 → 场景路由 Agent（识别问题类型）
              ├── IT 运维场景 → IT 运维专家 Prompt（强调排障逻辑）
              ├── HR 制度场景 → HR 制度专家 Prompt（强调政策条款）
              ├── 合规审计场景 → 合规专家 Prompt（强调法律依据）
              └── 通用场景   → 通用知识库 Prompt
```

**对外表述**：「KnowBrain 拥有 IT 运维专家、HR 政策专家、合规审计专家等多位 AI Agent，根据问题自动路由到最合适的专家回答。」

**对内实现**：

```java
// 本质是一个轻量分类器，不是真的多个 Agent 实例
String scene = sceneRouter.classify(question); // 快速文本分类
String systemPrompt = switch (scene) {
    case "it"    -> "prompts/expert-it.txt";     // IT 专家 Prompt
    case "hr"    -> "prompts/expert-hr.txt";     // HR 专家 Prompt
    case "legal" -> "prompts/expert-legal.txt";  // 合规专家 Prompt
    default      -> "prompts/rag-system.txt";    // 通用 Prompt
};
```

场景包已有的种子数据（categories + glossary + FAQ）正好可以用来训练/配置路由规则。基于关键词词典的分类器即可满足初期需求，不需要 LLM 分类（零额外成本）。

**预估工时**：后端 3 天（分类器 + 3 个专家 Prompt）+ 前端 1 天（显示当前活跃的专家 Agent）。

### 3.3 Agent Skills 演示包 ⭐⭐（投入低，Demo 利器）

给客户演示时，切换场景能看到 Agent 行为显著不同：

| 技能包 | 演示场景 | 差异化行为 |
|--------|---------|-----------|
| IT 运维 | 「VPN 连不上怎么办」 | 步骤式排障引导 + 追问日志检查 |
| HR 制度 | 「年假怎么算」 | 引用制度条款 + 工龄计算规则 |
| 合规审计 | 「数据出境需要什么审批」 | 法律条文引用 + 审批流程图 |

基于现有场景种子数据，给每个 Skill 配独立的 System Prompt + 示例问答即可。本质是 Prompt Engineering，零新代码。

**预估工时**：1 天（写 3 个 Prompt + 10 个 Demo 问题）。

### 3.4 MCP 外部接入 ⭐（投入中，扩展性强）

接一个外部系统（如 Jira/Confluence），让 Agent 不仅能搜文档，还能查外部数据：

- 「这个需求在 Jira 上的状态是什么」→ MCP 查 Jira
- 「Confluence 上有相关的设计文档吗」→ MCP 查 Confluence

Spring AI 已内置 MCP Client 支持。先做一个适配器验证流程，后续按需扩展。

**预估工时**：1-2 周（第一个适配器 + 前端展示外部数据来源标识）。

## 四、实施路线图

```
Phase A — 立即启动（2 周内见效）
├── 思考链可视化 —— 让已有 Agent 能力被看到
└── 场景路由 Agent —— 专家感知 + 零额外调用成本

Phase B — Demo 增强（1 周）
└── Agent Skills 演示包 —— 客户 Demo 差异化体验

Phase C — 按需扩展
├── 知识摘要 Agent —— 「整合这几份文档」
├── 知识巡检 Agent —— 「哪些制度已过期」
├── 新人向导 Agent —— 基于角色自动组装知识
└── MCP 外部接入 —— 连接企业现有系统

Phase D — 长期优化
└── 缩小 Context Recall 差距 → Agent 默认开启
```

## 五、与现有系统的关系

| 现有模块 | 如何配合 |
|---------|---------|
| 检索智能体（searchKnowledge） | 思考链可视化直接利用其多步检索过程 |
| 场景种子数据（categories/glossary/FAQ） | 场景路由 Agent 的分类词典基础 |
| FAQ 短路 | 不变，命中直接返回（不需要展示思考链） |
| Agent 开关（`RAG_AGENT_ENABLED`） | 默认关闭，思考链可视化在 Agent 路径下才激活 |
| OAuth2 / 手机号绑定 / 个人设置 | 不涉及，独立迭代 |

## 六、不做的红线

| 不做的事 | 原因 |
|---------|------|
| 通用 AI 能力（写作/翻译/代码/绘画） | 脱离知识库锚点，和 Dify/Coze/ChatGPT 正面竞争必输 |
| 真的多 Agent 并行协作框架 | 过度工程化，知识库场景不需要多 Agent 协商 |
| 知识图谱 | 混合检索 + Agent 多步检索已覆盖，投入大收益小 |
| 可视化 Agent Builder（拖拽式） | Dify 已经做得很好了，我们是知识库不是低代码平台 |
| 自进化/自动化工作流 | 企业场景更需要可控性，不是自治 |
| Agent V1（Function Calling + SQL 统计） | 与 Dashboard 高度重叠，价值不大（2026-07-19 已标记） |

---

## 七、总结

**一句话**：KnowBrain 的 Agent 战略 = 锚定知识全生命周期 + 低成本增强 AI 感知 + 不越界做通用平台。

**核心战术**：对外讲「多专家 Agent 协作」，对内用「场景路由 + Prompt 差异化」实现——感知层次上去了，架构复杂度不增加。

**成功标准**：客户 Demo 后说「这个 AI 和 ChatGPT 不一样，它真的懂我们的文档」——而不是「这东西和 ChatGPT 有什么区别」。
