## 一、执行摘要

### 1.1 项目概述

KnowBrain（知脑）是一个**纯私有化部署的企业智能知识库 RAG 平台**，技术栈为 Java 17 + Spring Boot 3.3 + Vue 3。核心能力是将企业文档投喂给 AI，员工用自然语言提问，AI 精准回答并标注原文出处。

### 1.2 核心结论

| 维度 | 评分 | 结论 |
|------|:----:|------|
| 市场时机 | ⭐⭐⭐⭐ | 企业 AI 私有化需求爆发，窗口期 12-18 个月 |
| 竞争差异化 | ⭐⭐⭐⭐ | Java + 纯私有化定位清晰，但需持续强化护城河 |
| 产品成熟度 | ⭐⭐⭐½ | MVP + IM 三平台完成，评测体系就绪，表格/Excel 基础可用，缺 Agent/LDAP |
| 商业模式 | ⭐⭐⭐ | 方向正确，定价/渠道策略待落地验证 |
| 团队匹配度 | ⭐⭐⭐⭐⭐ | 资深架构师 + 销售合伙人，互补性强 |
| **综合可行性** | **⭐⭐⭐⭐** | **可行，建议 6 个月内补齐关键缺口后正式商业化** |

### 1.3 一句话建议

> **以 "Java 企业版 Dify" 为定位，借助销售合伙人渠道获取前 5 个付费客户。企业 IM 集成（企微/钉钉/飞书）已完成，下一步聚焦表格识别、Agent 能力两大短板，在 2027 年上半年正式推向市场。**

### 3.3 KnowBrain 的核心差异优势

#### 优势 1：Java 技术栈——300 万 Java 开发者的 RAG 选择

```
中国开发者分布：
├── Java：~300 万（企业 IT 主力）
│   └── 当前 RAG 方案选择：几乎没有
├── Python：~150 万（AI/数据为主）
│   └── 可选方案：Dify, FastGPT, LangChain...
└── 其他：Node.js, Go, C# ...

KnowBrain 的机会：
  → 让 300 万 Java 开发者/团队不需要学 Python 就能部署 AI 知识库
  → 传统企业 IT（银行/保险/政务/制造业）Java 覆盖率 > 80%
  → 这些企业的运维/安全/采购部门天然信任 Java 方案
```

#### 优势 2：纯私有化——数据不出企业内网

| 场景 | KnowBrain | Dify Cloud | 飞书 AI |
|------|:---------:|:----------:|:------:|
| 金融合规（数据不出行） | ✅ | ❌ SaaS 不可用 | ❌ |
| 政府内网部署 | ✅ | ❌ | ❌ |
| 制造业图纸/工艺文档 | ✅ | ❌ 数据安全担忧 | ❌ |
| 律所合同/卷宗 | ✅ | ❌ | ❌ |

#### 优势 3：企业级权限管控——Dify 的明确短板

Dify 的 Knowledge 权限仅到应用级别，缺少文档级/部门级/空间级的细粒度权限。这是企业客户（尤其金融、政务）的刚需。

#### 优势 4：JDBC 标准生态集成

Java 生态天然对接企业现有基础设施——MySQL 主备、Redis 集群、LDAP 认证、企业监控（Zabbix/Prometheus），这些是 Python 方案的集成痛点。

### 3.4 竞争对手动向警示

| 竞品 | 最新动态 | 威胁等级 | 应对策略 |
|------|---------|:--------:|---------|
| **Dify** | A 轮融资，2025 年 Q3 推企业版 | 🔴 高 | 错位竞争：不做 Workflow 编排，专注知识库深度 |
| **FastGPT** | 开源社区活跃，OneAPI 生态绑定 | 🟡 中 | 强调 Java 集成便利性和权限管控 |
| **阿里百炼** | 2025 年强推私有化方案 | 🟡 中 | 阿里私有化定位大客户（百万级），不做 SME |
| **Coze/Dify Cloud** | SaaS 免费策略抢用户 | 🟢 低 | 目标客群不同——想要私有化的人不会选 SaaS |

### 7.2 缺口 1：表格识别 + OCR（基础已有，复杂场景待增强）

**当前已实现**：
- Qwen-VL 多模态 API：扫描件 PDF 自动检测 → PDFBox 渲染为 PNG → Qwen-VL OCR 提取文字
- POI 解析器：Excel (.xlsx/.xls) 表格原生提取
- `TableToMarkdown`：POI 表格 → Markdown 表格格式
- `ElementAwareChunker`：结构感知分块，识别标题/段落/表格/列表元素类型

**仍需增强**：
- 企业文档中 40%+ 关键信息在表格里（财务报表、制度条款对比表）
- 合并单元格 / 多级表头 / 扫描件中的表格 → 当前解析质量不够
- 提升方向：引入 MinerU / docling 专业文档解析引擎，LibreOffice 转 PDF 中间格式

**推荐方案**：
- 轻量增强：优化 POI 合并单元格处理 + Qwen-VL Prompt 增加表格结构保留指令
- 中长期：集成 MinerU（开源免费，表格提取准确率业界领先）作为第四解析器

### 7.3 缺口 2：企业 IM 集成 — ✅ 已完成

**状态：已于 2026-07 完成全部三平台集成。**

**实现成果**：
- ✅ 企微 Bot — HTTP 回调模式，AES-256-CBC 加解密 + SHA1 签名
- ✅ 钉钉 Bot — WebSocket Stream 模式，SDK 自动重连
- ✅ 飞书 Bot — WebSocket Stream 模式，智能双策略部门拉取
- ✅ IM 管理后台 — 部门映射 CRUD + 跨平台身份关联（`/admin/im`）
- ✅ 消息处理 — 去重/限流/用户映射/RAG 流水线三平台统一

**关键设计决策**：
- 企微走 HTTP 回调（`ImBotDispatcher` 统一调度），钉钉/飞书走 Stream 长连接（各自 Handler 内置 RAG）
- Stream 模式无需公网地址和 cloudflared 隧道，本地开发即可调试
- 飞书部门 API 采用 list→children 双策略智能降级，适配不同租户行为差异

详细设计见 [企业 IM 集成技术方案](./企业IM集成技术方案.md)。

### 7.4 缺口 3 详解：Agent/Function Calling — ✅ Phase 1 已完成

**实际交付方案（与早期规划不同）**：

早期规划了 SQL/统计/文档操作等多工具 Agent，经业界对标后调整为**检索智能体**——让 LLM 自主掌控检索策略，而非让 Agent 操作数据库。

**为什么是检索智能体而不是 SQL Agent**：
```
用户：「全职和外包远程办公政策有什么区别」
                  ↓
旧方案（SQL Agent）：LLM 生成 SQL → 查 DB → 无果（信息在文档里，不在数据库里）
                  ↓
新方案（检索智能体）：LLM 分析 → 分别搜索"全职远程政策""外包远程政策"
                     → 拿到双方信息 → 完整对比回答
```

**已交付能力**：
```
检索智能体 V1：
├── 多步检索：「VPN连不上导致远程桌面无法使用」→ 先搜 VPN 排障 → 补充远程桌面配置
├── 对比搜索：「全职和外包远程办公区别」→ 分别搜索两边 → 逐项对比
├── 歧义反问：「密码忘了」→ 反问"WiFi/邮箱/VPN 哪种密码？"
└── 反思自检：每次搜索后判断相关性/完整性 → 不够就换词重试（最多 3 轮）
```

**技术实现**：
- Spring AI Function Calling，零新依赖
- 单工具 `searchKnowledge`（包装 HybridSearchService），LLM 专注检索策略
- Prompt 内化反思（非独立 Refection 模型）
- 非流式 + 流式（SSE）双路径支持
- 默认关闭（`RAG_AGENT_ENABLED=false`），Feature Flag 渐进上线
- 异常自动降级到标准 RAG 管线

**Phase 2 规划**：MCP Server（对外暴露 KnowBrain 检索能力）+ 场景路由（IT vs HR 自动分流）
