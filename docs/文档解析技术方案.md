## 一、当前架构与问题分析

### 1.1 当前解析链路

```
upload → Tika 解析 → [扫描件?] → Qwen-VL OCR → SmartChunker → Embedding → Milvus
```

- **Tika 解析**：`DocumentServiceImpl.parseWithTika()`，使用 Spring AI 的 `TikaDocumentReader`（封装 Apache Tika）
- **扫描件检测**：Tika 提取文本 < 100 字符 → 触发 Qwen-VL OCR
- **OCR**：`PdfImageRenderer`（PDFBox 渲染页面为 PNG）+ `QwenVisionService`（DashScope 多模态 API）
- **分块**：`SmartChunker`，递归按分隔符切分（`\n\n` → `\n` → `。` → `！` → `？` → `；` → `，`），350 字/片，80 字重叠
- **向量化**：Spring AI EmbeddingModel dense + jieba 分词 sparse BM25

### 1.2 当前问题

| # | 问题 | 影响 | 商业化影响 |
|---|------|------|-----------|
| 1 | **表格结构丢失** | Tika 将表格转为平铺文本，行列关系全丢 | 搜索"Q3 营收"无法找到财报中的表格数据 |
| 2 | **不支持 xlsx/pptx** | `getFileType()` 仅识别 pdf/docx/txt/md | 企业客户高频格式无法上传 |
| 3 | **无元素类型分类** | 所有文本一视同仁，不区分标题/段落/表格/列表 | 分块质量差，检索精度低 |
| 4 | **分块策略盲切** | SmartChunker 按分隔符切分，Markdown 表格被拦腰截断 | 表格被拆散后无法检索 |
| 5 | **VLM 仅用于 OCR** | Qwen-VL Prompt 只是"提取文字"，未利用文档理解能力 | 输出是纯文本，非结构化 Markdown |
| 6 | **OCR 仅限 PDF** | `isScannedPdf()` 只对 `.pdf` 扩展名生效 | docx 中的扫描件图片无法处理 |
| 7 | **解析器无抽象** | 所有解析逻辑内联在 `DocumentServiceImpl` 私有方法中 | 无法扩展、测试、替换 |

### 2.3 业界核心思路

业界公认的文档解析最佳实践：

1. **深度学习布局检测**（非启发式规则）— 识别标题、段落、表格、图片、公式、页眉页脚
2. **VLM 文档→Markdown**（非 OCR 后处理）— 直接输出结构化 Markdown
3. **元素感知分块**（非盲切）— 表格/列表/代码块作为原子单元，不拆分
4. **Office 格式→PDF→统一管道**（非逐格式写逻辑）— LibreOffice headless 转换后走 PDF 管道

### 3.2 核心设计决策

| 决策点 | 选择 | 理由 |
|--------|------|------|
| 解析器架构 | **策略模式**（接口 + 优先级链） | 不改现有核心流程，易扩展 |
| PDF 引擎 | **Qwen-VL 文档→Markdown** | 已集成，Prompt 升级即可；大模型服务都是国内合规的 |
| PDF 降级 | Qwen-VL 不可用 → Tika | 离线环境/无 API Key 时自动降级 |
| Office 解析 | **Apache POI 原生提取** | Java 生态标准，纯内网可用 |
| 表格输出 | **Markdown 表格** | 嵌入分片管道，LLM 原生理解 |
| 未来增强 | MinerU（见第六章） | 断网部署/海量文档/公式密集时引入 |
| LibreOffice | **不引入**（见第六章） | Alpine 不可用；POI + Qwen-VL 已覆盖 Office 需求 |

### 3.3 解析器路由逻辑

```java
// ParserRouter
public ParsedDocument parse(byte[] fileBytes, String fileName) {
    String fileType = getFileType(fileName);

    return switch (fileType) {
        case "pdf"  -> qwenVlAvailable()
                       ? qwenVlParser.parse(fileBytes, fileName)  // Qwen-VL 主引擎
                       : tikaParser.parse(fileBytes, fileName);   // Tika 兜底

        case "docx",
             "xlsx",
             "pptx"  -> poiParser.parse(fileBytes, fileName);     // POI 原生

        case "txt",
             "md"    -> tikaParser.parse(fileBytes, fileName);    // 直接读取

        default     -> tikaParser.parse(fileBytes, fileName);     // 未知格式兜底
    };
}
```

## 四、核心组件设计

### 4.1 DocumentParser 接口

```java
package com.knowbrain.document.service.parser;

public interface DocumentParser {
    /** 支持的扩展名列表，如 ["pdf", "docx"] */
    List<String> supportedExtensions();

    /** 解析文件，返回结构化 Markdown 文本 */
    ParsedDocument parse(byte[] fileBytes, String fileName) throws ParseException;

    /** 优先级（数字越小越优先，Tika 兜底 = 999） */
    default int priority() { return 100; }
}

/** 解析结果 */
public record ParsedDocument(
    String markdown,           // 结构化 Markdown（含表格/标题/列表）
    List<TableBlock> tables,   // 提取的表格（可选，用于增强检索）
    int pageCount,             // 页数
    ParseMetadata metadata     // 解析元数据（引擎、耗时、置信度）
) {}
```

### 4.2 PoiParser — Office 格式解析

**docx 处理策略**：
- XWPFDocument 读取段落文本（含标题层级 → `#`/`##`）
- 表格 → `XWPFTable` 逐单元格读取 → `TableToMarkdown` 转换
- 合并单元格 → 检测 `gridSpan`/`vMerge` → 填充占位符
- 嵌入图片 → 提取二进制 → 上传 MinIO → 替换为 `![alt](minio_url)`
- SmartArt/文本框/图表 → 标记为「含复杂元素」→ 路由到 QwenVlParser

**xlsx 处理策略**：
- 逐 Sheet 遍历 → `## Sheet名称`
- 每行 → Markdown 表格行
- 合并单元格 → 检测 `CellRangeAddress` → 填充相邻单元格
- 公式单元格 → 优先取缓存值，回退公式表达式
- 数字格式 → 应用 `DataFormatter` 获取显示值
- 大表（> 50 行）→ 独立存储，不参与普通分块

**pptx 处理策略**：
- 逐幻灯片 → `# 幻灯片 N: 标题`
- 文本框 → Z-order 排序 → 自然阅读顺序
- 表格（PPT 内嵌）→ Markdown 表格
- 演讲者备注 → 引用块 `> 备注: ...`
- 检测到大量图表 → 降级为 QwenVlParser

### 4.4 QwenVlParser — PDF 主引擎

**Prompt 升级**（关键改动）：

```
当前 Prompt（仅 OCR）：
"请提取图片中的所有文字内容，保持原文的段落结构和表格格式，
 不要添加任何说明或注释。"

升级后 Prompt（文档理解）：
"将文档页面转为 Markdown 格式。
 要求：
 1. 表格用 | 列 | 列 | 格式，表头和数据行用 |---| 分隔
 2. 标题按层级使用 #、##、###
 3. 列表项使用 - 或 1.
 4. 保持原文的阅读顺序
 5. 如果页面中没有文字，回复「无文字内容」
 6. 不要添加任何说明或注释"
```

**调用方式**（复用现有 QwenVisionService）：
- PDF → `PdfImageRenderer` 逐页渲染 → Qwen-VL 逐页 Markdown 化
- 复杂 docx → 整体转图片 → Qwen-VL 识别
- 每页结果用 `\n\n---\n\n` 分隔

### 4.5 ElementAwareChunker — 结构感知分块

**核心改动**：在原有 SmartChunker 基础上增加 Markdown 结构感知层。

```
输入: Markdown 文本
  ↓
Step 1: 结构解析 — 识别 Markdown 节点
  ├── 表格块 (|...|)       → 标记为 ATOMIC_TABLE
  ├── 代码块 (```)          → 标记为 ATOMIC_CODE
  ├── 标题 (#/##/###)       → 标记为 HEADING
  ├── 列表项 (-/* / 1.)     → 标记为 LIST_ITEM
  └── 普通段落              → 标记为 PARAGRAPH
  ↓
Step 2: 原子化 — 不可拆分的块作为一个单元
  ├── ATOMIC 块 → 如果超长，截断 + "（内容过长，已省略中间部分）"
  └── PARAGRAPH 块 → 走原有 SmartChunker 递归切分
  ↓
Step 3: 上下文增强
  └── 每个 chunk 前缀插入最近的上层标题（如 "## 财务摘要"）
  ↓
输出: List<String> chunks（含结构上下文的原子化分块）
```

### 6.1 MinerU — 纯本地 SOTA PDF 引擎

#### 6.1.1 是什么

[MinerU](https://github.com/opendatalab/MinerU) 是上海 AI 实验室开源的文档解析工具，核心能力：

- DocLayout-YOLO 布局检测（标题/段落/表格/公式/图片）
- UniMERNet 公式识别（LaTeX 输出，SOTA）
- 内置 OCR（所有页面渲染为图片后统一分析，不区分文本/扫描件）
- 阅读顺序恢复（ML 模型，解决多栏混排）
- OmniDocBench v1.6 评分 **95.69**，超过 GPT-4o 和 Gemini 2.5 Pro

#### 6.1.2 当前为什么不引入

| 理由 | 说明 |
|------|------|
| Qwen-VL 已满足需求 | 已集成，Prompt 升级后直接输出 Markdown，种子客户阶段完全够用 |
| MVP 阶段文档量小 | 日均 < 50 页，Qwen-VL API 成本可忽略（约 ¥0.02/页） |
| 部署成本 > 收益 | MinerU 镜像 ~3GB（含模型），需 GPU 才能发挥速度优势，CPU 模式比 Qwen-VL 还慢 |
| 过早优化 | 先验证产品 PMF，再优化基础设施 |

#### 6.1.3 什么情况下需要引入

**必须引入（阻断业务）**：

| 场景 | 判断标准 | 为什么 |
|------|---------|--------|
| 断网/离线部署 | 客户明确提出"系统不能访问外部网络" | Qwen-VL 调 DashScope API 需要外网，MinerU 纯本地运行 |
| 数据不出境合规 | 金融/政务客户要求数据不出企业内网 | DashScope API 数据传输到阿里云，合规风险 |

**建议引入（ROI 为正）**：

| 场景 | 判断标准 | 为什么 |
|------|---------|--------|
| 海量文档 | 日均上传 > 500 页且持续增长 | Qwen-VL API 月度成本开始显著（500页/天 × 30天 × ¥0.02 ≈ ¥300/月，持续增长） |
| 速度敏感 | 用户上传后需秒级完成解析 | Qwen-VL ~3s/页，MinerU GPU 0.21s/页，14 倍差距 |
| 公式密集 | 客户文档含大量数学/化学公式 | UniMERNet 公式→LaTeX 准确率远超 VLM 通用能力 |

**不需要引入**：

| 场景 | 为什么 |
|------|--------|
| 制度/合同/报告类文档为主 | Qwen-VL 输出质量已足够 |
| 日均 < 50 页 | API 成本忽略不计 |
| 种子验证阶段（< 5 个客户） | 产品打磨优先，基础设施够用就行 |

#### 6.1.4 引入方案（预研已完成）

**部署**：Docker Sidecar 容器，独立服务，AGPL 协议不传染 Java 主代码。

```yaml
# docker-compose.yml（在 services 下新增）
mineru:
  image: opendatalab/mineru:latest
  container_name: knowbrain-mineru
  restart: unless-stopped
  ports:
    - "9987:9987"
  environment:
    MINERU_MODE: pipeline              # pipeline(CPU) 或 vlm(GPU)
  volumes:
    - mineru-data:/data
  profiles:
    - full                              # 可选组件，默认不随主服务启动
```

**Java 侧**：新增 `MinerUClient implements DocumentParser`，REST 调用 MinerU API。路由逻辑改动：

```java
// ParserRouter 中 PDF 路由改为：
if (mineruClient.isHealthy()) {
    return mineruClient.parse(pdfBytes, fileName);  // MinerU 优先
}
if (qwenVlAvailable()) {
    return qwenVlParser.parse(pdfBytes, fileName);  // Qwen-VL 备选
}
return tikaParser.parse(pdfBytes, fileName);        // Tika 兜底
```

**评估指标**：引入前后跑第七章的评测数据集，对比解析准确率和速度。

### 6.2 LibreOffice — Office 格式→PDF 转换

#### 6.2.1 是什么

[LibreOffice](https://www.libreoffice.org/) 是开源办公套件，`--headless` 模式可以将 docx/xlsx/pptx 转为 PDF。业界 RAG 系统的常见做法（Azure Document Intelligence、Unstructured.io 均采用"先转 PDF 再统一解析"的策略）。

#### 6.2.2 当前为什么不引入

| 理由 | 说明 |
|------|------|
| Alpine 不可用 | `apk search libreoffice` 返回空——Alpine（musl libc）与 LibreOffice（依赖 glibc）不兼容 |
| 镜像急剧膨胀 | 切换为 Ubuntu 基础镜像 + 安装 LibreOffice → Java 镜像从 ~200MB 膨胀到 ~800MB |
| 容器稳定性差 | LibreOffice headless 在容器中已知问题：字体缺失→中文乱码、内存泄漏→OOM、`soffice` 僵尸进程需 `killall` 清理、单个 `soffice` 进程不支持并发转换 |
| 有更简单的替代 | Qwen-VL（PDF）+ POI（Office）已覆盖所有场景，没必要引入"先转 PDF 再解析"这种绕路方案 |

#### 6.2.3 什么情况下需要引入

**必须引入（POI 确实搞不定）**：

| 场景 | 判断标准 | 为什么 POI 不行 |
|------|---------|----------------|
| pptx 解析质量被客户投诉 | 付费客户明确反馈 PPT 内容丢失/顺序错乱，且 Qwen-VL 截图识别也无法满足 | pptx 无"文档流"概念，POI 按 Z-order 近似阅读顺序只是启发式 |
| 复杂 docx 版面 100% 精确还原 | 需要合同/标书的版面精确还原（非结构化文本提取），且 POI 丢失了关键信息 | SmartArt、文本框、嵌入对象 POI 不可达 |
| 批量格式统一转换需求 | 客户需要"上传任意格式，都能转成统一格式存档" | POI 只能读不能渲染 |

**不需要引入**：

| 场景 | 为什么 |
|------|--------|
| 常见制度文档、报告、合同 | POI 已经能提取段落 + 表格 + 标题层级 |
| Excel 数据表 | POI 直接读单元格值更精确——转 PDF 反而依赖 OCR |
| 客户没有 pptx 需求 | 绝大部分企业知识库以 PDF + docx 为主 |

#### 6.2.4 引入方案（预研）

> **推荐 Sidecar 模式**。不推荐将 LibreOffice 嵌入 Java 镜像（Alpine 不可用 + 稳定性风险）。

```yaml
# docker-compose.yml（在 services 下新增）
libreoffice:
  image: linuxserver/libreoffice:latest
  container_name: knowbrain-libreoffice
  restart: unless-stopped
  volumes:
    - libreoffice-tmp:/tmp               # 共享临时目录
  profiles:
    - full                                # 可选组件
```

Java 侧新增 `LibreOfficeConverter`，流程：

```
Office 文件 → LibreOffice headless → PDF → QwenVlParser（复用 PDF 管道）
```

```java

    public byte[] convertToPdf(byte[] input, String ext) {

### 6.3 决策速查表

| 问题 | 答案 |
|------|------|
| 客户要求断网部署？ | → **引入 MinerU**（PDF 解析刚需） |
| 客户要求离线也能解析 Office 文件？ | → 暂不处理（POI 纯本地可用，只是质量略低于转 PDF 再 VLM） |
| PDF 解析太慢/太贵？ | → **引入 MinerU**（0.21s/页 + 免费） |
| pptx 解析质量被投诉？ | → **引入 LibreOffice**（转 PDF 后走 Qwen-VL） |
| 文档含大量公式？ | → **引入 MinerU**（UniMERNet SOTA） |
| 日均 < 50 页且客户无特殊要求？ | → **保持现状**（Qwen-VL + POI + Tika） |

## 七、评测标准

### 7.1 评测数据集

准备 30 份真实企业文档，覆盖以下类别：

| 类别 | 数量 | 示例 |
|------|:----:|------|
| 制度/规范类 PDF（原生文字） | 5 | 员工手册、考勤制度 |
| 表格密集型 PDF | 5 | 财务报表、价格清单 |
| 扫描件 PDF | 5 | 盖章合同、老文件扫描 |
| 制度/报告类 docx | 5 | 项目报告、制度说明 |
| 含表格 docx | 3 | 岗位对照表、制度条款表 |
| Excel 数据表 | 4 | 配置参数表、HR 统计表 |
| PPT 演示文稿 | 3 | 培训课件、项目汇报 |

### 7.2 评测指标

| 指标 | 目标 | 评测方法 |
|------|:----:|---------|
| 表格提取准确率（单元格级） | ≥ 90% | 人工标注 vs 提取结果逐格对比 |
| 表格行列数正确率 | ≥ 95% | 自动对比表格维度 |
| 标题层级保留率 | ≥ 95% | 人工标注 vs 提取结果 |
| 扫描件 OCR 字符准确率 | ≥ 95% | CER（字符错误率） |
| 分块表格完整性 | 100% | 表格不应跨 chunk 拆分 |
| 端到端检索命中率 | ≥ 85% | 30 个测试问题 × Top-5 检索命中率 |

### 7.3 评测流程

```
准备测试文档 → 解析入库 → 设计测试问题（每条问题标注预期检索的 chunk）
→ 执行检索 → 计算 Top-1/Top-3/Top-5 命中率 → 生成评测报告
```
