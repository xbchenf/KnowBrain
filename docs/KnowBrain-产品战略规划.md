# EICS V3 — 企业智能知识大脑 产品战略规划

> 版本：V1.0 | 日期：2026-07-01 | 作者：架构师 + AI 辅助

---

## 一、战略背景

### 1.1 EICS V2.0 回顾

V2.0 完成了一次成功的**技术架构升级**（Python/Rasa → 纯 Java 状态机 + LLM 增强），但产品定位上存在模糊地带：

| 问题 | 现状 |
|------|------|
| 定位分散 | 同时想做 RAG 问答 + 工单系统 + 坐席聊天，三个模块深度都不够 |
| 竞争乏力 | 每个单独模块都比不过专注做那一件事的产品（工单不如 Jira，客服不如智齿） |
| 知识库最弱 | RAG 是三个模块中产品化程度最低的，但恰恰是市场机会最大的 |
| 缺乏差异化 | 功能对标大厂 SaaS，却没有大厂的生态和渠道 |

### 1.2 核心洞察

> **大厂的"通用知识平台"都在云上。私有化部署 + AI 知识检索，大厂几乎不做——利润薄、交付重、不符合 SaaS 订阅制商业模式。这是个人/小团队的机会窗口。**

同时，2025-2026 年企业 AI 市场验证了三个事实：

1. **RAG 是当前 LLM 落地成功率最高的模式**——不凭空生成，始终基于客户自己的文档，幻觉可控
2. **企业愿意为"数据不外传"买单**——中大型企业对私有化部署有刚性需求（合规 + 安全）
3. **AI 问答的 ROI 直接可量化**——新人培训周期缩短、老员工重复答疑减少、知识检索时间从小时级降到秒级

## 二、产品定位

### 2.1 一句话定位

> **私有化部署的企业智能知识大脑——把公司所有文档投喂进去，员工用自然语言提问，AI 精准回答 + 溯源原文。**

### 2.2 一套代码，两层能力

**重要：这不是两套系统，是一套代码、一个部署包。「场景应用」不是独立应用，而是同一平台加载不同预配置数据后呈现出的效果。**

打个比方：WordPress 装好之后，装「电商主题」还是「企业官网主题」——代码是同一套，换的是模板和配置。IT 运维助手同理——装上 IT 知识模板和术语词典，平台就变成了「IT 运维助手」。

```
     ┌──────────────────────────────────────────┐
     │        knowbrain-server.jar（后端）         │
     │                                          │
     │  ┌────────────────────────────────────┐  │
     │  │  场景配置层（种子数据，非代码）       │  │
     │  │  ├─ IT 运维模板（首发）              │  │
     │  │  ├─ HR 制度模板（规划中）            │  │
     │  │  └─ 法务合规模板（规划中）           │  │
     │  │  → 上传不同种子数据包即可切换场景     │  │
     │  └──────────────┬─────────────────────┘  │
     │                 │ 纯粹的数据驱动           │
     │  ┌──────────────▼─────────────────────┐  │
     │  │  平台引擎层（通用能力，核心代码）     │  │
     │  │  ├─ 文档接入 → 解析 → 分块 → 向量化 │  │
     │  │  ├─ 混合检索 → Rerank → LLM 生成   │  │
     │  │  ├─ 权限管控 → 溯源 → 反馈闭环      │  │
     │  │  └─ 管理后台 + 问答门户             │  │
     │  └────────────────────────────────────┘  │
     └──────────────────────────────────────────┘
                         │
           Docker Compose 一键部署（同一堆栈）
```

- **平台引擎层**：核心代码，不绑定任何行业，部署后传什么文档就答什么
- **场景配置层**：预配置知识分类模板 + 术语词典 + 高频问答对 + 对话流定义，全部是**数据，不是代码**

### 2.3 首发场景：IT 运维智能助手

选择 IT 运维作为第一个场景切入，原因：

| 维度 | 理由 |
|------|------|
| 决策链短 | IT 部门自己就能决定采购，不需要 CIO/CTO 审批 |
| 痛点极痛 | 任何 50 人以上公司的 IT 部门都在被重复问题淹没 |
| 知识结构化 | IT 技术文档、故障手册、配置指南恰恰最适合 RAG |
| 竞品空白 | ServiceNow 太重太贵，飞书/钉钉知识库 AI 能力弱，纯私有化的 IT 知识问答几乎没有 |
| 团队熟悉 | EICS V2.0 积累的工单 + SLA 领域知识可复用 |

## 三、目标用户与核心场景

### 3.1 用户画像

| 角色 | 典型用户 | 核心诉求 |
|------|---------|---------|
| **知识消费者** | 普通员工、新入职员工 | 「VPN 怎么配置」「年假怎么申请」「XX 项目的技术方案在哪」——快速找到答案 |
| **知识生产者** | IT 运维、HR、部门主管 | 把经验文档化→让 AI 替自己回答重复问题 |
| **知识管理者** | IT 管理员、知识库管理员 | 文档上传/分类/权限管控、知识质量监控、未命中问题分析 |

### 3.2 核心使用场景

```
场景 1：新人入职自助
  新员工：「公司 VPN 怎么配置？」
  AI → 精准返回配置指南，附带截图和步骤
  效果：不用追着老员工问，入职首日即可自助解决常见 IT 问题

场景 2：故障排查引导
  员工：「电脑蓝屏了，错误代码 0x0000007B」
  AI → 识别错误代码 → 给出排查步骤（安全模式→驱动检查→硬件诊断）
  效果：80% 常见故障 AI 引导自助解决，减少 IT 工单量

场景 3：制度政策查询
  员工：「产假可以休多少天？需要什么材料？」
  AI → 返回公司考勤制度相关条款 + 申请流程 + 所需材料清单
  效果：HR 减少 50% 以上的重复政策咨询

场景 4：技术文档检索
  开发：「用户中心微服务的接口文档在哪？」
  AI → 返回接口文档链接 + 关键接口说明
  效果：跨团队协作不再靠「问对人」，直接问 AI

场景 5：历史经验复用
  运维：「上次数据库死锁怎么解决的？」
  AI → 检索历史工单/故障报告 → 返回当时的排查过程和解决方案
  效果：专家离职经验不丢失，故障处理不再从零开始
```

## 四、核心竞争力

### 4.1 差异化壁垒

| 能力 | 为什么是壁垒 |
|------|------------|
| **私有化部署** | 数据不出企业内网，满足合规要求。大厂不做是因为交付成本高 |
| **混合检索 + 重排序** | 语义检索（Milvus）+ 关键词检索（ES）+ Reranker 精排，比纯向量检索召回率提升 20%+ |
| **文档级权限管控** | 不同部门/角色看到不同文档，A 部门的薪酬制度不能让 B 部门搜到 |
| **溯源到原文段落** | 每个答案精确标注来源文档 + 段落位置，用户可以自己验证，解决信任问题 |
| **反馈闭环** | 用户点「有用/无用」→ 知识质量评分 → 驱动知识库更新 |
| **场景预配置** | 首发 IT 运维场景内置知识分类模板、术语词典、高频问题库，开箱即用 |

### 4.2 为什么大厂不会来做

| 大厂模式 | 本产品 |
|---------|--------|
| SaaS 订阅制（数据在云端） | 私有化部署（数据在客户机房） |
| 标准化交付（零定制） | 支持轻度场景定制 |
| 按用户数/Token 计费 | 一次性部署费用 + 年维护费 |
| 大客户优先（100 万+ 合同） | 中型企业也能负担（5-20 万） |

## 六、技术架构

### 6.1 技术选型

| 层 | 技术 | 说明 |
|----|------|------|
| 后端框架 | Java 17 + Spring Boot 3.3.x | 企业级标准，EICS V2.0 可复用 |
| AI 编排 | Spring AI 1.0+ | 统一 LLM 调用抽象 |
| LLM | Qwen-Max / DeepSeek-V3 | DashScope/DeepSeek API，OpenAI 兼容协议 |
| 向量数据库 | Milvus 2.4 | HNSW 索引，高性能语义检索 |
| 关键词检索引擎 | Elasticsearch 8.x | 可选组件，用于 BM25 关键词检索增强 |
| 关系数据库 | MySQL 8.0 | 文档元数据、用户权限、反馈数据 |
| 缓存 | Redis 7 | 问答缓存（高频问题秒级响应）、会话状态 |
| 对象存储 | MinIO | 原始文档存储 |
| 文档解析 | Apache Tika 2.x | PDF/Word/TXT/Markdown 多格式解析 |
| 前端 | Vue 3 + Element Plus | EICS V2.0 前端可复用 |
| 部署 | Docker Compose | 一键部署，单机或集群 |

### 6.2 核心检索架构

```
用户问题
   ↓
┌──────────────────────────────┐
│  查询预处理                    │
│  ├─ 问题改写（LLM 补全上下文）  │
│  ├─ 术语映射（VPN→虚拟专用网络）│
│  └─ 敏感词过滤                 │
└──────────┬───────────────────┘
           ↓
┌──────────────────────────────┐
│  混合检索                      │
│  ├─ 语义检索（Milvus 向量）    │ ← 捕捉语义相似但用词不同的问题
│  └─ 关键词检索（BM25/ES）     │ ← 精确匹配专业术语/编号/代码
└──────────┬───────────────────┘
           ↓
┌──────────────────────────────┐
│  粗排 → 精排                   │
│  ├─ 粗排：向量距离 + BM25 分数  │
│  └─ 精排：Reranker 重排序模型   │ ← Cross-encoder，精准度提升显著
└──────────┬───────────────────┘
           ↓
┌──────────────────────────────┐
│  LLM 生成                      │
│  ├─ Prompt 拼装（Top-K 文档块） │
│  ├─ 答案生成 + 引用标注         │
│  ├─ 置信度判断（低于阈值→"未找到"）│
│  └─ 敏感词/幻觉后检             │
└──────────┬───────────────────┘
           ↓
    答案 + 溯源 → 返回用户
```

### 6.3 权限过滤流程

```
用户提问
   ↓
获取用户角色 + 所属部门（JWT Token）
   ↓
构建权限过滤条件：user_dept IN document_visible_depts
   ↓
Milvus 检索时注入过滤表达式（Milvus 支持标量过滤）
   ↓
仅返回用户有权查看的文档块
```

## 七、商业模式

### 7.1 定价策略

| 版本 | 价格 | 内容 |
|------|------|------|
| **社区版** | 免费 | 核心检索能力，单机部署，最多 1000 文档，社区支持 |
| **专业版** | ¥19,800/年 | 不限文档数、权限管控、混合检索、管理后台、邮件支持 |

### 7.2 为什么年费制

- 社区版免费 → 降低试用门槛，企业先用起来
- 专业版年费 → 覆盖后续迭代研发成本
- 企业版年费 → 对中大型企业仍是极低成本（一个 IT 坐席月薪就能覆盖一年）

## 八、MVP 开发路线图

### 阶段 1：平台核心引擎（第 1-2 周）

```
目标：文档进去 → AI 答案出来

□ 项目脚手架搭建（Spring Boot 3.3 + MyBatis-Plus + Milvus + Redis + MinIO）
□ 文档上传 API + MinIO 存储
□ Tika 文档解析（PDF/Word/TXT/MD → 纯文本）
□ 智能分块引擎（递归分块，语义边界感知，可配置块大小和重叠）
□ Milvus 向量化入库（Embedding 模型调用 + 批量写入）
□ 检索 → 生成流水线（向量检索 → Top-K 拼接 Prompt → LLM 生成 → 溯源标注）
□ 基础 Web 问答界面
```

### 阶段 2：权限 + 管理后台（第 3-4 周）

```
目标：可管理、可控制

□ 用户认证（JWT + BCrypt，复用 EICS V2.0）
□ 文档空间模型（Space → Document → Chunk 三层）
□ 空间级权限（Owner/Editor/Viewer + 部门可见范围）
□ 检索时权限过滤（Milvus 标量过滤表达式）
□ 管理后台 UI（文档上传/分类/删除/搜索）
□ 使用统计看板（问答量/命中率/热门问题 Top 10）
```

### 阶段 3：IT 运维场景包（第 5-6 周）

```
目标：IT 部门开箱即用

□ IT 知识分类模板（网络/设备/软件/权限/安全）
□ IT 术语词典（50+ 术语别名映射）
□ 高频 IT 问题标准答案库（预设问答对，不走检索直接返回）
□ 历史工单导入工具（Excel/CSV → 自动向量化）
□ 检索质量调优（针对 IT 场景调 Embedding + 分块参数 + Reranker 阈值）
□ Demo 环境搭建 + 演示数据集准备
```

### 阶段 4：打包发布（第 7-8 周）

```
目标：5 分钟部署、10 分钟 demo 说服客户

□ Docker Compose 编排（eics-server + MySQL + Redis + Milvus + MinIO + Web）
□ 一键初始化脚本（数据库建表 + 种子数据 + Milvus Collection 创建）
□ 安装部署文档 + 视频教程
□ Demo 演示脚本（标准演示流程：上传文档 → 提问 → 看溯源 → 看统计）
□ 社区版 GitHub Release
```

## 十、风险与对策

| 风险 | 概率 | 影响 | 对策 |
|------|------|------|------|
| LLM API 延迟/不可用 | 中 | 高 | 高频问题缓存（Redis），热点问题预计算，支持切换多个 LLM 供应商 |
